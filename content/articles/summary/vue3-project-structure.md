## 为什么很多 Vue 项目写到后面会变慢

原因通常不是 Vue 本身，而是项目结构让复杂度不断回流到页面组件。页面一开始只是模板加请求，后来逐渐加上：

- 权限判断
- 接口格式转换
- 弹窗状态
- 表单校验
- 业务规则
- 埋点逻辑
- 缓存与重试

最后一个页面文件会同时承担视图、数据、状态、流程和规则。这样一来，任何改动都可能牵动整块逻辑。

## 我更推荐的拆分方法

Vue 3 时代，组合式 API 提供了很好的拆分工具，但前提是你知道“该按什么拆”。我通常会先按下面四类来判断：

### 1. 页面协调逻辑

页面负责组织数据流和交互顺序。例如当前搜索条件是什么、当前页需要展示哪些模块、切换 tab 时谁要刷新。

这部分逻辑可以留在页面里，因为它天然和页面路由绑定。

### 2. 业务能力逻辑

这类逻辑更适合抽到 composable 或 service 中。比如：

- 搜索文章
- 获取精选内容
- 保存筛选条件
- 统一处理文章模型

这些能力不是某个页面独有的，就不应该写死在页面里。

### 3. 呈现组件

像卡片、目录、头部导航、搜索面板这种内容，本质上是 UI 单元。它们最好是输入确定、输出确定、对业务细节知道得越少越好。

### 4. 数据源适配

这是很多项目最容易漏掉的一层。比如服务端返回字段是 `updated_at`，页面展示用的是 `updatedAt`，这一步转换最好不要在页面里发生。

## 组合式函数不是“什么都往里放”

Vue 3 提供了 composable 之后，很多项目会把页面里的逻辑直接搬到 `useXxx` 文件里，但这不等于结构改善。

一个 composable 如果同时做了这些事：

- 请求接口
- 维护多个 UI 状态
- 监听 DOM
- 做数据格式转换
- 处理错误提示

那它只是把混乱从页面移到了另一个文件。

更好的原则是：一个 composable 只封装一类变化。

例如：

```ts
useArticleSearch()
useFeaturedArticles()
useSiteProfile()
```

每个 composable 的边界都应该从名字上能看出来。

## 页面为什么要尽量“变薄”

变薄不是为了好看，而是为了控制风险。

当页面足够薄时，你可以很快回答下面几个问题：

- 出错的是数据层还是 UI 层
- 替换接口时要改哪几处
- 做国际化时哪些文案需要抽离
- 新增一个筛选条件会影响哪里

这类问题如果答不出来，说明结构已经开始失控。

## 一个常见但代价很高的坏味道

最常见的问题是“页面知道太多”。

比如页面里同时存在：

```ts
const articles = await $fetch('/api/articles')
const featured = articles.filter(...)
const grouped = transformCategory(articles)
const canEdit = roles.includes('editor')
```

这意味着页面自己承担了请求、筛选、转换和权限判断。随着逻辑增长，这个页面会越来越难维护。

更合理的做法是让页面读到的就是相对稳定的结果。

## 对个人项目尤其重要的一点

个人项目很容易被“快速实现”驱动，因为没有多人评审和协作约束。也正因为如此，结构问题更容易被忽视。

如果你希望这个站点不是三个月后就废弃的 demo，而是一个持续累积的技术资产，那么页面和数据层的边界越早明确越好。

## 结语

Vue 3 项目结构设计的核心，不是目录长什么样，而是谁应该知道什么。

页面只负责页面，组件只负责展示，数据层只负责提供稳定能力。只要这件事守住，项目就不会轻易失控。
