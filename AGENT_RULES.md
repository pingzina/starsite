# StarSite Agent 规则文档

这份文档是给后续参与本项目的 Agent 或协作者看的。目标很简单：避免每次修改都把项目结构搞乱。

## 1. 基本原则

- 不要随意推翻现有目录结构
- 不要把页面直接耦合到底层内容来源
- 不要把简单内容写死在页面组件里
- 修改前优先理解当前结构，再动手
- 保持工程化，不做一次性拼凑

## 2. 内容相关规则

- 技术文章优先放在 `content/articles/`
- 长文内容使用 Markdown，不要改成大段 JSON
- frontmatter 字段保持统一，不要私自新增一堆不兼容字段
- 图片、音频、演示资源优先通过 `media` 字段接入
- 草稿内容使用 `draft: true`

## 3. 配置相关规则

- 个人资料、分类、导航配置优先放在 `app/config/site.ts`
- 文案国际化放在 `i18n/locales/`
- 环境变量统一通过 `runtimeConfig` 暴露
- 不要在页面中直接读取 `process.env`

## 4. 页面开发规则

- 页面只负责展示和交互
- 数据读取优先走 `/api/*` 或 composable
- 不要在页面里直接写内容查询细节
- 文档页、搜索页、首页保持一致的视觉语言
- 新增页面时优先复用现有组件

## 5. 服务端和数据访问规则

- 内容查询统一集中在 `server/utils/content.ts`
- 如果未来接后端接口，优先替换这里，不要直接改每个页面
- `/api/articles`、`/api/search`、`/api/profile` 这些接口签名尽量保持稳定
- 新增内容字段时，要同步更新：
  - `content.config.ts`
  - `types/content.ts`
  - `server/utils/content.ts`

## 6. 样式规则

- 保持“简洁、有秩序、偏官方文档站”的视觉方向
- 不要改成花哨营销页风格
- 优先修改：
  - `assets/css/main.css`
  - 组件内部 Tailwind 类
- 不要引入第二套大型 UI 框架，除非明确需要

## 7. 国际化规则

- 当前是“界面双语，内容单语优先”
- 不要求所有文章都必须有英文版本
- 新增界面文案时，必须同步补全：
  - `i18n/locales/zh-CN.json`
  - `i18n/locales/en-US.json`

## 8. 媒体资源规则

- 静态资源优先放在 `public/` 下
- 路径使用站点绝对路径，例如 `/media/demo.svg`
- 不要把大型二进制资源随便塞进仓库
- 演示音频或图片优先用小体积示例

## 9. 代码修改规则

- 修改时优先小步调整，不要无理由大改
- 如果只是改内容，不要碰页面逻辑
- 如果只是改样式，不要顺手重构数据层
- 保持 TypeScript 类型可通过
- 修改完成后至少验证：
  - `npm run build`
  - `npm run typecheck`

## 10. 部署规则

- 当前首发目标是静态部署
- 改动时要注意 `npm run generate` 仍然可用
- 如果涉及 GitHub Pages，要注意 `NUXT_APP_BASE_URL`
- `robots.txt`、`rss.xml`、`sitemap.xml` 不要无故删掉

## 11. 允许扩展的方向

下面这些方向可以继续增强，但要按现有架构扩展：

- 全文搜索
- 评论系统
- 数据统计
- 标签页
- 专题页
- 项目作品页
- 接口化内容源
- 后台管理或 CMS

## 12. 禁止事项

- 禁止把 Markdown 内容全迁到页面里
- 禁止把分类、导航和个人资料散落到多个页面硬编码
- 禁止未验证就升级一批核心依赖
- 禁止为了一个小需求引入重型框架
- 禁止在没有必要的情况下破坏静态导出能力

## 13. 给未来 Agent 的一句话

这个项目的核心不是“堆功能”，而是保持：

- 内容容易维护
- 结构清晰
- 视觉稳定
- 后续好扩展

任何修改都应该围绕这四点展开。
